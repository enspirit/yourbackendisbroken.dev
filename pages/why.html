Let's improve all backends

As we continuously worked on software for years, we came to the conclusion that many teams could improve their software quality through better design & black-box testing of backend components.

A backend is broken when …

  1. It becomes harder and harder to maintain and make evolve,
  2. or breaks its clients when released,
  3. or throws 500 errors when something is unexpected,
  4. or exposes technical details on errors,
  5. or has recurrent performance problems,
  6. or requires looking at the code to discover the services.

Hidden costs of broken backends

  1. Your team is 10 times slower than it could be.
  2. Your backend will eventually be rewritten from scratch.
  3. A deadly security nightmare waits around the corner.
  4. You miss an opportunity to think clearly about your software.

Testing is incomplete in most teams

  1. Unit tests are fined-grained, and may be too brittle.
  2. Integration testing is technically hard, most teams skip it.
  3. End-to-end testing is often slow, and focuses on best case scenarios.
  4. Functional black-box testing is ... generally unknown.

Web services offer an opportunity to improve quality

  1. Web services correspond to high-level software operations.
  2. They offer a natural space for stable design.
  3. Being public by nature, they call for robustness & security analysis.
  4. The HTTP protocol is standard and technology agnostic.

We provide an open-source framework where…

  1. Web services are specified in a friendly, yet formal way.
  2. Checking if they meet their specification is as easy as using curl.
  3. PRE/POST contracts provide systematic testing coverage.
  4. A specification can also be used for documentation, API mocking, code generation, etc.

It's different from existing alternatives

  1. Our first focus is on testing & quality, not documentation or UI tools.
  2. Specifications are independent from the implementation, not derived from it.
  3. Therefore they are human-centric, not machine-centric.
  4. A spec is a spec, not a usage scenario.
